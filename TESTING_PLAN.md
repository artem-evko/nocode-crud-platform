# Комплексный план тестирования No-Code CRUD Platform

> **Адресат:** AI-агент (ИИ-тестировщик).
> **Цель:** Полное покрытие всей функциональности платформы и генерируемых приложений.
> **Среда:** Windows, Docker Desktop, PostgreSQL, Node.js 20+, Java 21+.

---

## Предусловия запуска

Перед выполнением любого тест-кейса убедись, что:

1. **Docker Desktop** запущен и работает.
2. Сеть `nocode-crud-platform_nocode-network` существует (`docker network ls`).
3. PostgreSQL-контейнер платформы запущен: `docker compose up -d` в корне проекта.
4. Backend платформы запущен: `powershell -ExecutionPolicy Bypass -File .\start-platform.ps1` (порт 8082).
5. Frontend платформы запущен: `cd platform-frontend && npm run dev` (порт 5173).
6. URL платформы: `http://localhost:5173`.
7. Креды по умолчанию: `admin` / `admin`.

## Раздел 1: Аутентификация платформы

### TC-1.1: Успешный вход в платформу
- **Шаги:**
  1. Открыть `http://localhost:5173/login`.
  2. Ввести логин `admin`, пароль `admin`.
  3. Нажать кнопку «Войти».
- **Ожидание:** Происходит редирект на `/projects`. Отображается имя пользователя `admin` в шапке. HTTP-ответ `POST /api/auth/login` — статус 200.

### TC-1.2: Вход с неверным паролем
- **Шаги:**
  1. Открыть `http://localhost:5173/login`.
  2. Ввести логин `admin`, пароль `wrong_password`.
  3. Нажать кнопку «Войти».
- **Ожидание:** На экране появляется ошибка «Неверное имя пользователя или пароль». Остаёмся на `/login`. HTTP-ответ — 401.

### TC-1.3: Вход с пустыми полями
- **Шаги:**
  1. Открыть `http://localhost:5173/login`.
  2. Оставить оба поля пустыми.
  3. Нажать кнопку «Войти».
- **Ожидание:** Браузерная HTML5-валидация блокирует отправку (поля `required`). Запрос на сервер НЕ отправляется.

### TC-1.4: Регистрация нового пользователя
- **Шаги:**
  1. Открыть `http://localhost:5173/register`.
  2. Ввести уникальный логин (например, `testuser_<timestamp>`) и пароль `Test1234`.
  3. Нажать кнопку «Зарегистрироваться».
- **Ожидание:** Ответ 200. Переход на страницу логина или автоматический вход. Пользователь затем может залогиниться с новыми кредами.

### TC-1.5: Регистрация с уже занятым логином
- **Шаги:**
  1. Открыть `http://localhost:5173/register`.
  2. Ввести логин `admin`, пароль `Test1234`.
  3. Нажать кнопку «Зарегистрироваться».
- **Ожидание:** Ответ 400. Сообщение содержит текст «Username is already taken».

### TC-1.6: Выход из платформы (Logout)
- **Шаги:**
  1. Войти как `admin`.
  2. На странице `/projects` нажать кнопку «Выйти».
- **Ожидание:** Редирект на `/login`. HTTP-запрос `POST /api/auth/logout` — 200. Повторный запрос `GET /api/auth/me` без сессии — 401.

### TC-1.7: Защита роутов от неавторизованного доступа
- **Шаги:**
  1. Выйти из системы (или открыть инкогнито-окно).
  2. Перейти напрямую на `http://localhost:5173/projects`.
- **Ожидание:** Происходит автоматический редирект на `/login`. Страница проектов НЕ отображается.

## Раздел 2: Управление проектами (CRUD)

### TC-2.1: Создание проекта с минимальными данными
- **Шаги:**
  1. Войти в платформу.
  2. На странице `/projects` нажать «Новый Проект».
  3. Заполнить обязательные поля:
     - Название: `TestProject`
     - Group ID: `com.test`
     - Artifact ID: `test-api`
     - Базовый пакет: (должен сгенерироваться автоматически как `com.test.testapi`)
     - Версия: `1.0.0`
  4. Оставить опции «Авторизация (JWT)» и «Фронтенд» выключенными.
  5. Нажать «Создать проект».
- **Ожидание:** Проект создан. Происходит редирект на страницу моделирования `/projects/<uuid>/modeler`. Поле `basePackage` автосгенерировалось корректно.

### TC-2.2: Создание проекта со включённой авторизацией и фронтендом
- **Шаги:**
  1. Нажать «Новый Проект».
  2. Заполнить: Название: `FullProject`, Group ID: `com.full`, Artifact ID: `full-app`.
  3. Включить чекбокс «Авторизация (JWT)».
  4. Включить чекбокс «Фронтенд (React + Vite)».
  5. Нажать «Создать проект».
- **Ожидание:** Проект создан. Поля `authEnabled: true`, `generateFrontend: true` сохранены (можно проверить через `GET /api/projects`).

### TC-2.3: Валидация полей при создании проекта
- **Шаги:** Попытаться создать проект с каждым некорректным значением по отдельности:
  - **Group ID:** `COM.Example` (заглавные буквы) → ожидание: ошибка валидации.
  - **Group ID:** `123com` (начинается с цифры) → ожидание: ошибка валидации.
  - **Artifact ID:** `My App` (пробелы) → ожидание: ошибка валидации.
  - **Artifact ID:** `MyApp` (заглавные буквы) → ожидание: ошибка валидации.
  - **Base Package:** `com.example.123` (цифра в начале сегмента) → ожидание: ошибка валидации.
  - **Пустое название:** (пустая строка) → ожидание: ошибка «Project name is required».
- **Ожидание:** Для каждого случая — HTTP 400 с сообщением об ошибке. Проект НЕ создан.

### TC-2.4: Редактирование проекта
- **Шаги:**
  1. На странице `/projects` навести мышку на карточку проекта.
  2. Нажать иконку «Редактировать» (карандаш).
  3. Изменить название проекта на `UpdatedProject`.
  4. Нажать «Сохранить изменения».
- **Ожидание:** Модальное окно закрылось. Карточка отображает новое название `UpdatedProject`.

### TC-2.5: Удаление проекта (с подтверждением)
- **Шаги:**
  1. Навести на карточку проекта и нажать иконку «Удалить» (корзина).
  2. В модальном окне подтверждения нажать «Удалить».
- **Ожидание:** Toast-уведомление «Проект успешно удален». Карточка исчезает из списка. HTTP DELETE возвращает 200.

### TC-2.6: Отмена удаления проекта
- **Шаги:**
  1. Навести на карточку проекта и нажать «Удалить».
  2. В модальном окне подтверждения нажать «Отмена».
- **Ожидание:** Модальное окно закрывается. Проект остаётся в списке.

### TC-2.7: Пустой список проектов
- **Шаги:**
  1. Удалить все существующие проекты.
  2. Или зарегистрировать нового пользователя и войти.
- **Ожидание:** Отображается блок «Проектов пока нет» с кнопкой «Создать Проект». Кнопка работает.

### TC-2.8: Автогенерация Base Package
- **Шаги:**
  1. Открыть форму «Новый проект».
  2. Ввести Group ID: `com.example`.
  3. Ввести Artifact ID: `my-app`.
  4. Не трогать поле Base Package.
- **Ожидание:** Base Package автоматически заполнилось значением `com.example.myapp` (дефисы убраны, lowercase).

## Раздел 3: Моделирование данных (Modeler)

### TC-3.1: Добавление сущности
- **Шаги:**
  1. Открыть страницу моделирования существующего проекта.
  2. Нажать кнопку «Добавить сущность» в панели инструментов.
- **Ожидание:** На холсте ReactFlow появился новый узел `NewEntity` с одним полем `id` типа `UUID`. Узел можно перетащить.

### TC-3.2: Переименование сущности
- **Шаги:**
  1. Кликнуть на заголовок сущности `NewEntity`.
  2. Ввести новое имя `Product`.
  3. Нажать Enter или кликнуть вне поля.
- **Ожидание:** Узел отображает имя `Product`. Валидация пройдена (нет красной подсветки).

### TC-3.3: Невалидное имя сущности
- **Шаги:**
  1. Кликнуть на заголовок сущности.
  2. Ввести имя с пробелом `My Entity` или с цифры `1Entity` или с кириллицей `Сущность`.
  3. Нажать Enter.
- **Ожидание:** Узел подсвечивается красной рамкой и / или красным свечением. ОР: регулярное выражение `/^[a-zA-Z][a-zA-Z0-9_]*$/` не проходит.

### TC-3.4: Добавление поля в сущность
- **Шаги:**
  1. Нажать кнопку «Добавить поле» внутри узла сущности.
- **Ожидание:** Появляется новое поле с дефолтным именем `newField`, типом `String`, required = false.

### TC-3.5: Редактирование поля (имя, тип, обязательность)
- **Шаги:**
  1. Изменить имя поля на `title`.
  2. Изменить тип через dropdown на `Integer`.
  3. (Через панель свойств, если доступно) включить `required`.
- **Ожидание:** Поле отображается с обновлёнными значениями. Иконка типа соответствует `Integer` (значок `#`).

### TC-3.6: Удаление поля
- **Шаги:**
  1. Навести на поле в узле сущности.
  2. Нажать кнопку «×» (удалить).
- **Ожидание:** Поле удалено из списка полей узла.

### TC-3.7: Поддержка всех типов данных
- **Шаги:** Добавить поле и последовательно выбрать каждый тип из dropdown:
  - `String`, `Integer`, `Long`, `Boolean`, `LocalDate`, `OffsetDateTime`, `UUID`, `Double`, `BigDecimal`.
- **Ожидание:** Каждый тип успешно выбирается и сохраняется. Иконка типа соответствует категории.

### TC-3.8: Связывание сущностей (Edges)
- **Шаги:**
  1. Добавить две сущности: `Category` и `Item`.
  2. Протянуть линию (edge) от правого handle сущности `Category` к левому handle сущности `Item`.
- **Ожидание:** На холсте появляется связь. При сохранении модели эта связь попадает в `edges` и компилируется в relation `ONE_TO_MANY`.

### TC-3.9: Настройка RBAC ролей
- **Шаги:**
  1. В узле сущности раскрыть секцию «Права доступа (RBAC)».
  2. В поле «Чтение (Роли)» ввести `ROLE_USER, ROLE_ADMIN`.
  3. В поле «Создание (Роли)» ввести `ROLE_ADMIN`.
  4. Оставить «Обновление» и «Удаление» пустыми.
- **Ожидание:** Значения сохраняются в модели. При генерации кода — контроллер содержит `@PreAuthorize("hasAnyAuthority('ROLE_USER', 'ROLE_ADMIN')")` на методе GET и `@PreAuthorize("hasAnyAuthority('ROLE_ADMIN')")` на методе POST.

### TC-3.10: Сохранение модели
- **Шаги:**
  1. Добавить 2 сущности с полями и связью.
  2. Нажать кнопку «Сохранить модель».
- **Ожидание:** Toast «Модель успешно сохранена!». HTTP PUT `/api/projects/<id>` — 200. Поле `specText` содержит JSON со spec и `_flow` (nodes + edges).

### TC-3.11: Загрузка ранее сохранённой модели
- **Шаги:**
  1. Сохранить модель (TC-3.10).
  2. Уйти на страницу проектов (`/projects`).
  3. Кликнуть на карточку проекта для возврата к моделированию.
- **Ожидание:** Все ранее добавленные сущности, поля, связи и позиции узлов восстановлены из `specText`.

### TC-3.12: Несколько сущностей с несколькими полями каждого типа
- **Шаги:**
  1. Создать сущность `Order` с полями: `orderDate` (LocalDate), `total` (BigDecimal), `isPaid` (Boolean), `description` (String).
  2. Создать сущность `Customer` с полями: `name` (String), `age` (Integer), `email` (String).
  3. Соединить `Customer → Order`.
  4. Сохранить модель.
- **Ожидание:** Модель сохранена без ошибок. Spec содержит 2 entities с корректными типами полей и relation.

## Раздел 5: Генерация и скачивание кода

### TC-5.1: Скачивание кода (Backend Only)
- **Шаги:**
  1. Создать проект без фронтенда.
  2. Добавить 1 сущность с 2 полями. Сохранить модель.
  3. Нажать «Скачать код».
- **Ожидание:** Браузер скачивает ZIP-файл. Внутри — Maven-проект с `pom.xml`, `Application.java`, `application.yml`, сгенерированные: Entity, Repository, Controller, Liquibase changelog.

### TC-5.2: Скачивание кода (Backend + Frontend)
- **Шаги:**
  1. Создать проект с включенным фронтендом.
  2. Добавить 1 сущность с полями. Сохранить.
  3. Нажать «Скачать код».
- **Ожидание:** ZIP содержит директории `backend/` и `frontend/`. Frontend содержит `package.json`, `src/App.tsx`, сгенерированные `EntityList.tsx`, `EntityForm.tsx`, `api.ts`.

### TC-5.3: Скачивание кода (Backend + Auth + Frontend)
- **Шаги:**
  1. Создать проект с `authEnabled: true` и `generateFrontend: true`.
  2. Добавить 1 сущность. Сохранить.
  3. Нажать «Скачать код».
- **Ожидание:** ZIP содержит `backend/src/.../security/` с файлами: `SecurityConfig.java`, `JwtUtil.java`, `JwtAuthenticationFilter.java`, `AuthController.java`, `User.java`, `UserRepository.java`, `UserDetailsServiceImpl.java`. Frontend содержит `LoginPage.tsx`, `RegisterPage.tsx`, `authStore.ts`.

### TC-5.4: Скачивание кода без сохранения модели (edge case)
- **Шаги:**
  1. Создать проект. **НЕ** добавлять сущности и **НЕ** сохранять.
  2. Нажать «Скачать код».
- **Ожидание:** Либо toast с ошибкой «Ошибка при генерации кода. Вы сохранили модель?», либо скачивается минимальный скелет проекта (зависит от реализации).

## Раздел 14: Edge Cases и граничные условия

### TC-14.1: Сущность без полей (кроме id)
- **Шаги:**
  1. Добавить сущность.
  2. Удалить все поля кроме `id`.
  3. Сохранить и попытаться развернуть.
- **Ожидание:** Валидатор «Доктор No-Code» показывает ошибку: «В сущности не добавлено ни одного поля».

### TC-14.2: Сущности с дублирующимися именами
- **Шаги:**
  1. Добавить 2 сущности.
  2. Назвать обе `Product`.
  3. Попытаться развернуть.
- **Ожидание:** Валидатор выдаёт ошибку: «Обнаружены сущности с одинаковым именем».

### TC-14.3: Длинное название сущности (50+ символов)
- **Шаги:** Назвать сущность строкой из 60 латинских букв.
- **Ожидание:** Имя принимается (если соответствует regex). Узел может усечь отображение (truncate), но данные валидны.

### TC-14.4: Большое количество сущностей (10+)
- **Шаги:**
  1. Добавить 10 сущностей с 3-5 полями каждая.
  2. Сохранить модель.
  3. Развернуть проект.
- **Ожидание:** Модель сохраняется без ошибок. Развёртывание проходит, Liquibase создаёт 10 таблиц. Фронтенд содержит 10 страниц.

### TC-14.5: Спец-символы в значениях полей сгенерированного приложения
- **Шаги:** В сгенерированном приложении создать запись с `name`: `<script>alert("xss")</script>`.
- **Ожидание:** Значение сохранено как строка. При отображении — React безопасно экранирует HTML (XSS не исполняется).

### TC-14.6: Очень длинное значение поля
- **Шаги:** Создать запись с `description` длиной 10 000 символов.
- **Ожидание:** Запись сохранена (если нет ограничения max в валидации). Отображается в таблице (возможно с обрезкой).

### TC-14.7: Конкурентное редактирование (Optimistic Locking)
- **Шаги:**
  1. Открыть проект в двух вкладках.
  2. В первой вкладке изменить модель и сохранить.
  3. Во второй вкладке (со старой версией) попытаться сохранить.
- **Ожидание:** HTTP 409 Conflict с сообщением: «The project was modified in another window. Please reload and try again.»
